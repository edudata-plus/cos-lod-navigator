# 学習指導要領LODナビゲーター UI仕様書（案）

## 1. 目的

本システムは、学習指導要領LODの構造を、現場の教員が直感的に理解・探索できるようにするためのWebブラウザ型UIである。
このUIを用いて、検索によって細目へ直接到達した利用者が、その細目を学習指導要領本文全体の文脈の中で理解できるよう支援する。
従来のLODデータが持つURI、コード、親子関係、周辺文脈などの構造情報を、教員等の利用文脈に合わせて「検索」「周辺文脈ナビゲーション」「本文表示」を主要機能とする。

比較表示やつながり表示などは今後の拡張候補とし、第1段階の必須機能には含めない。

## 2. 想定する対象

### 2.1 想定利用者

主な利用者は、小学校・中学校・高等学校の教員、教育委員会関係者、教材開発者、教育学研究者、学校図書館司書などである。
特に、学習指導要領の本文を単に検索するだけでなく、本文の章節構造から、学校種・教科・学年・領域間を横断した、または、領域内に存在するつながりを把握したい利用者を対象とする。

### 2.2 対象とする学習指導要領と学校種別

なお、本ツールが対象とする学習指導要領は、「現行学習指導要領 平成29・30・31年 (2017・18・19) 改訂」とする。

学習指導要領LODにおいて以下のURIが付与された学習指導要領細目のみを対象とする。
- https://w3id.org/jp-cos/Elementary/2017
- https://w3id.org/jp-cos/LowerSecondary/2017
- https://w3id.org/jp-cos/UpperSecondary/2018

また、対象とする学校種別は、小学校、中学校、高等学校（ただし、いわゆる専門教科の範囲は除く）の3つとする。

幼稚園教育指導要領および特別支援学校の以下のURIが付与された細目は第2段階で対象とする。
- https://w3id.org/jp-cos/Kindergarten/2017
- https://w3id.org/jp-cos/KindergartenDeptSNES/2017
- https://w3id.org/jp-cos/ElementaryAndLowerSecondaryDeptSNES/2017
- https://w3id.org/jp-cos/UpperSecondaryDeptSNES/2019

各細目からは cs:courseOfStudy プロパティで接続されている。

## 3. 画面構成

画面は大きく以下の領域で構成する。

### 3.1 ヘッダー

ヘッダーには、システム名、タグライン、ナビゲーターのバージョン番号を配置する。

### 3.2 メインエリア

メインエリアは、ヘッダー下にすべての主要な画面をおさめる領域である。

メインエリア内には、グリッド列の3カラム構成とし、左から以下の領域を配置する：

1. 検索ボックス・フィルタ
2. 本文・検索結果・学びのつながり領域
3. 周辺文脈領域

画面サイズにあわせて可変とし、画面幅が十分に得られない時は、画面幅全体にあわせて上から順に表示すること。


### 3.3 検索ボックス・フィルタ領域

検索ボックス、主要操作ボタンを配置する。
利用者は任意のキーワードを入力し、学習指導要領LOD内の該当項目を検索できる。

検索ボックス下に、検索キーワードが思いつかない利用者向けに、検索例を示す目的で、「防災」「情報活用」「データ」「探究」「環境」「表現」などのキーワードを表示し、それぞれリンク化して、リンクをクリックするだけで検索結果を示せるようにすること。

また、検索対象を絞り込むため、以下の絞り込みフィルタを設ける。

* 学校種
* 教科
* 科目
* 学年

フィルタは複数条件を組み合わせて利用できるものとする。

また、検索語を入れなくともブラウジングが開始できるよう、学校種の選択だけはできるようにしておきたい。
※学校種を選択すると、その学校種にかかわるすべての細目が検索結果に出てくるようにする。

なお、フィルタ絞り込みの機能は、ラベル表記の一致による絞り込みとはせず、URIを用いた絞り込みとすること。
学校種、教科、科目、学年による絞り込みであれば、それぞれ細目の cs:school, cs:subjectArea, cs:subject, cs:grade で与えられたURIを用いて絞り込むこと。

### 3.4 周辺文脈ナビゲーション

#### 3.4.1 機能の位置づけ

周辺文脈ナビゲーションは、学習指導要領LODの階層関係とその構造を反映する主要領域である。
この機能は、学習指導要領本文を「読む」ためではなく、「位置づけを理解する」ためのナビゲーションとして位置付ける。そのため、利用頻度が高い学校種・教科・学年・大項目まではツリー形式とし、それ以下の細かな階層は一覧表示によって俯瞰できるようにする。

本システムにおける本文表示は、単に同一キーワードを含む項目を推薦する機能ではなく、選択された細目が学習指導要領全体のどの階層に位置づけられているかを示すための、階層的ナビゲーション機能である。

現行の学習指導要領本文は、基本的に章・節・学年・内容・細目の順に、線形に読むことを前提として構成されている。本システムでは、この線形構造を利用しつつ、任意の細目を起点として、その上位階層、同階層の前後項目、近接する関連細目を一画面内で確認できるようにする。

#### 3.4.2 表示内容

本文詳細ページを表示した際の周辺文脈ナビゲーション領域には、選択中の細目を中心として、以下の階層を上から順に表示する。

* 学習指導要領
* 章
* 節
* 教科
* 学年
* 目標または内容
* 大項目
* 中項目
* 小項目
* 細目

表示はツリー形式とする。選択中の細目は強調表示し、同じ階層にある前後の細目もあわせて表示する。これにより、利用者は当該細目がどの文脈で記述されているか、また前後にどのような学習内容が配置されているかを把握できる。

たとえば、細目「(イ) 観察・調査したり地図などの資料で調べたりして，白地図などにまとめること。」を表示する場合、周辺文脈領域には以下のような階層を表示する。

小学校学習指導要領
→ 第2章 各教科
→ 第2節 社会
→ 第2 各学年の目標及び内容
→ 〔第3学年〕
→ 2 内容
→ (1) 身近な地域や市区町村の様子について
→ ア 次のような知識及び技能を身に付けること
→ (ア) 身近な地域や自分たちの市の様子を大まかに理解すること
→ (イ) 観察・調査したり地図などの資料で調べたりして，白地図などにまとめること
→ イ 次のような思考力，判断力，表現力等を身に付けること
→ (2) 地域に見られる生産や販売の仕事について

この表示では、選択中の細目だけでなく、その直前・直後に位置する項目も表示する。特に、同一大項目内の「ア」「イ」のような構成や、同一内容内の「(1)」「(2)」の並びを保持することで、本文を前から順に読んだ場合の文脈を失わないようにする。

周辺文脈領域の目的は、利用者に対して「この細目の周辺には何があるか」を示すことである。そのため、単なる上位階層の表示にとどまらず、次の3種類の関係を同時に示す。

1. 上位関係
   当該細目が属する章・節・教科・学年・内容・大項目を表示する。
2. 同階層関係
   当該細目と同じ親項目に属する兄弟項目を表示する。
3. 前後関係
   学習指導要領本文上で当該細目の直前・直後に位置する項目を表示する。

この周辺文脈表示により、利用者は検索結果から直接細目に到達した場合でも、当該細目の前後関係や階層的位置づけを失わずに理解できる。
なお、選択細目はそれとわかるように強調表示する。

兄弟項目とは、同じ親項目を持つ項目である。
前後項目とは、学習指導要領本文上の出現順において、選択細目の直前または直後に現れる項目である。
兄弟項目と前後項目は一致する場合もあるが、親項目をまたいで前後に続く場合は区別して扱う。

また、上下、前後、兄弟階層間で、省略した細目がある場合は、そこに「続きを表示」というリンクを付けて、それをクリックすると、そこに省略されていた細目が15件まで追加取得できるような機能を用意すること。

#### 3.4.3 生成ルール

周辺文脈ナビゲーションは、選択細目を起点として生成する。

1. 選択細目のURIを取得する。
2. schema:isPartOf または逆向きの schema:hasPart により、上位項目を再帰的に取得する。
3. ルート項目から選択細目までの祖先系列を構成する。
4. 選択細目の親項目を特定し、その親項目に属する子項目を本文順に取得する。
5. 選択細目の前後に位置する項目を特定する。
6. 必要に応じて、親項目の前後に位置する兄弟項目も取得する。
7. 取得した項目をツリー形式に整形し、選択細目を強調表示する。

LOD上の親子関係が取得できる場合は、`schema:hasPart` 等の関係を用いて階層構造を構成する。
節番号階層が存在する場合は、`sectionNumberHierarchy` を用いて表示順と階層深度を補助する。

なお、大項目より細かな中項目・小項目・細目のレベルは深くなりすぎるため、周辺文脈ナビゲーションで一つずつ展開する手間なく、全体を一覧表示し、その場でながめられるようにすること。

#### 3.4.4 表示ルール

周辺文脈ナビゲーションでは、選択細目を中心として、以下の範囲を表示する。

1. ルートから選択細目までの祖先項目
2. 選択細目の親項目に属する全ての子項目
3. 選択細目の親項目の前後に位置する兄弟項目
4. 必要に応じて、選択細目の祖父母項目に属する主要な兄弟項目

各項目は折りたたみ可能とし、選択された項目は本文表示領域に反映する。

なお、初期表示では、上記4条件にあてはまる項目群はすべて展開状態とする。
ただし、表示項目数が25件を超える場合、初期状態では上記の4条件の上から順に25件を超過するまで採用した項目のみを展開表示しておくが、それら以外は折りたたんで表示し、ユーザによる展開操作によって表示できるようにする。

また、上記の初期表示における展開ルールにかかわらず、前後項目のうち、前3つ・後3つの項目まではすべて展開状態とする。
上記にあてはまらない、それ以外の祖父母系列の関連項目や離れて隣接する項目は折りたたみ状態とする。

なお、選択中の細目は、背景色・左罫線・ラベル等により明確に強調表示する。
また、選択細目の祖先項目も弱い強調表示とし、現在位置までの経路が視覚的に追えるようにする。

本文上の順序は qb:order -> rdf:value の値により決定する。

### 3.5 検索結果表示領域

検索結果表示領域では、検索結果を項番順に示す。
検索結果は簡潔に保つために、学校種、教科・科目・学年（あれば）に加えて、本文を示すこととする。

検索結果では、本文をスニペット状に表示するとともに、検索語などヒット箇所を分かりやすくハイライト表示すること。

なお、表示エリアを効果的に使うため、3.5節、3.6節、3.7節の領域はメインエリアの中央列を占め、それぞれをタブ切り替えで使えるようにすること。
初期状態では、検索結果領域を表示するものとし、検索結果に示された項目を選択した場合には、シームレスに本文表示タブに切り替えること。
また、新たに検索実行された場合には、検索結果表示タブに自動的に切り替わることとする。

### 3.6 本文表示領域

本文表示領域では、選択された項目の内容を表示する。
表示項目は以下とする。

* 学校種
* 教科
* 科目
* 学年
* 階層
* 本文
* 細目コード
* URI
* 親項目
* 周辺文脈

教員にとって読みやすいよう、本文を中心に表示し、URIやコードなどの技術的情報は補助情報として配置する。

また、学習指導要領LODのリンクは一貫したデザインで表示して、適宜、詳細はLODサイトを確認できるようにすること。

### 3.7 学びのつながり表示

学びのつながり表示では、選択された項目について、同一キーワードを含む項目だけでなく、学年/教科間の接続関係や学校種を越えた接続関係を時系列に表示する。

学びのつながりは、小学校から高等学校までの系統性を俯瞰するためのビューであり、本文の上下関係とは独立したナビゲーション機能とする。

学びのつながりの表示には、検索語を用いて、学校種・学年・教科を横断して同一語を含む項目を時系列に表示する。
高度な概念対応、意味的類似、教材との対応づけは今後の拡張とする。

たとえば「防災」「情報活用能力」「データ活用」などのテーマについて、小学校から中学校、高等学校への接続を確認できるようにする。

今後、第2弾以降の開発において、選択細目内の代表語2～3などを自動抽出するなどしてつながりを表示する方法論を開発する。

## 4. 検索機能

検索機能は、教員が自然な語句で学習指導要領本文を探索できることを目的とする。
検索対象は以下とする。

* 本文
* 教科名
* 科目名
* 細目コード
* URI
* 解説テキスト

検索結果には、該当本文だけでなく、その項目が属する階層を必ず表示する。
これにより、利用者は検索語がどの文脈で使われているかを把握できる。

なお、解説テキストは一部の細目にのみ存在するため、存在する場合のみ扱うこと。
* 小学校学習指導要領・理科および社会科・第5, 6学年のみ

また、検索結果から細目を選択した場合、本文表示領域には当該細目を表示し、周辺文脈ナビゲーション領域では当該細目までの階層を自動的に展開する。
検索語がある場合は、本文表示および検索結果スニペット内で検索語をハイライト表示して、検索キーワードとの関係も分かりやすく示すこと。


## 5. データ取得仕様

本システムはGitHub Pages上で動作する静的Webアプリケーションとして実装する。
サーバーサイド処理は持たず、ブラウザから学習指導要領LODの公開SPARQLエンドポイントに問い合わせる。

取得対象となる主な情報は以下である。

* URI
* ラベル
* 本文
* 学校種
* 教科
* 科目
* 学年
* 細目コード
* 親子関係
* 節番号階層
* 周辺文脈
* 項番情報

基本的に、1回のSPARQLで取得し、必要に応じて追加取得する。また、取得した情報はキャッシュし、再度の問い合わせ回数を少なくすること。

たとえば、画面表示時に必要最小限を取得し、兄弟取得は必要時のみ取得するなどの工夫を行うこと。

SPARQL取得に失敗した場合、システムは以下を表示する。

1. 接続失敗メッセージ
2. 再取得ボタン


## 6. UI設計方針

本UIでは、LODの技術的構造をそのまま見せるのではなく、教員の利用文脈に翻訳して提示する。

重視する設計方針は以下である。

1. 検索結果の位置が分かること
2. 一画面で文脈を把握できるようにする
3. 本文を読むより構造を理解できることを重視する
4. 検索結果から文脈へ戻れること
5. 検索より閲覧を重視する
6. 学習内容の上下関係が分かること
7. 学年間・学校種間のつながりが分かること
8. 他教科との関係が分かること
9. 同じ画面内で前後関係へ移動できるようにする
10. 細目コードを必要に応じて確認できること
11. RDFの概念を利用者に意識させない
12. URIは必要時のみ表示する


## 7. 画面遷移仕様

### 7.1 初期画面

初期画面は、全体構成を一度に把握でき、その後の操作を通じても変化がないよう、ヘッダエリアの直下にメインエリアを配置した状態で開始する。

また、初期状態では、検索実行されていないため、検索結果はゼロ件から開始する。

ただし、検索フィルタにおいて、学校種別は、「小学校」「中学校」「高等学校」の3つを選択可能とし、いずれかが選択された場合は、全細目の中から、当該学校種に絞り込んだ検索結果を表示すること。

### 7.2 画面遷移

利用者は以下のように、タブなどを切り替えながら、シームレスに詳細情報を確認できるような画面遷移とすること。

```
トップ
   ↓
検索結果
   ↓
本文表示＋周辺文脈（前後・上下・学びのつながり）
   ↓
本文表示 または 検索結果
```
